# Integration Testing Design

**Status:** Approved
**Date:** 2026-02-19

## Goal

Build an automated integration test suite that tests the one-shot-build plugin phase-by-phase using real Claude Code CLI invocations against a synthetic project. When tests fail, an autonomous fix agent diagnoses the issue, patches the plugin code, and re-runs the test.

## Requirements

- **Phase-by-phase** tests: init, profile-data, define-epics, execute-plan
- **Synthetic toy project** as the test fixture (deterministic, cheap)
- **Real Claude Code CLI** invocations (not mocked)
- **Local developer machine** execution
- **Fix-then-retry** autonomous remediation on failure
- **Two-tier assertions**: structural checks (hard gate) + content quality (advisory warnings)
- **Python + pytest** test harness

## Architecture Overview

```
run_all.py (orchestrator)
  │
  ├─ fixtures/synthetic/        Deterministic test data + analyst answers
  │
  ├─ agents/
  │   ├─ setup_agent.py         Generates prompt templates per phase
  │   ├─ responder_agent.py     Live answers to unexpected questions
  │   └─ fix_agent.py           Diagnose-fix-verify on failures
  │
  ├─ assertions/
  │   ├─ structural.py          Hard-gate file/structure checks
  │   └─ quality.py             Advisory content checks
  │
  ├─ Happy-path tests (sequential in chained mode):
  │   ├─ test_init.py
  │   ├─ test_profile_data.py
  │   ├─ test_define_epics.py
  │   └─ test_execute_plan.py
  │
  └─ Edge-case tests (parallel via pytest-xdist):
      ├─ test_circuit_breakers.py
      ├─ test_resume_recovery.py
      └─ test_guards.py
```

## Synthetic Toy Project

A single CSV file (`fixtures/synthetic/customers.csv`) with ~100 rows and 6 columns:

| Column | Type | Range |
|---|---|---|
| `customer_id` | int | unique |
| `age` | int | 18-80 |
| `tenure_months` | int | 1-60 |
| `monthly_spend` | float | 10-500 |
| `support_tickets` | int | 0-10 |
| `churned` | bool | target variable |

Generated by `fixtures/synthetic/generate.py` with a fixed random seed. The CSV is checked into the repo for speed.

**Modeling task:** Predict `churned` from the other features via logistic regression. Deliberately trivial — we're testing the harness workflow, not modeling quality.

**Pre-built analyst answers** (`fixtures/synthetic/analyst-context.yaml`):

```yaml
project_name: "integration-test-churn"
business_objective: "Predict customer churn from demographic and behavioral features"
target_variable: "churned"
data_paths: ["customers.csv"]
exclude_columns: ["customer_id"]
domain_notes: "This is synthetic test data. No special domain constraints."
epic_preferences: "Keep it simple - data loading, model training, evaluation"
```

## Test Harness Architecture

### Directory structure

```
integration_tests/
├── conftest.py
├── fixtures/
│   └── synthetic/
│       ├── generate.py
│       ├── customers.csv
│       └── analyst-context.yaml
├── agents/
│   ├── setup_agent.py
│   ├── responder_agent.py
│   └── fix_agent.py
├── assertions/
│   ├── structural.py
│   └── quality.py
├── test_init.py
├── test_profile_data.py
├── test_define_epics.py
├── test_execute_plan.py
├── test_circuit_breakers.py
├── test_resume_recovery.py
├── test_guards.py
└── run_all.py
```

### Key pytest fixtures (`conftest.py`)

- `test_project_dir` — creates a fresh temp directory, initializes a git repo, copies in `customers.csv`. Cleaned up after the test.
- `claude_runner` — wraps `subprocess.Popen` for launching `claude` with the plugin loaded. Handles stdin/stdout streaming, timeout management, and output capture.
- `phase_output` — captures the full session transcript + filesystem state after each phase for assertions.

### Test isolation

Each phase test gets its own temp directory by default, using pre-built fixture files for prior phase outputs. A `--chained` flag enables a shared directory where each phase feeds into the next. In chained mode, if a phase fails, subsequent phases are skipped.

## The Setup Agent

Runs before each phase test. Reads the skill definition for that phase and the analyst context YAML, then generates a prompt template via a haiku API call.

Prompt templates are cached by file hash of the skill definition. Only regenerated when the skill changes.

**Generated prompts per phase:**

| Phase | Skill read | Key answers baked in |
|---|---|---|
| init | `harness-init/SKILL.md` | project name |
| profile-data | `profile-data/SKILL.md` | data paths, business objective, target, exclusions, sign-off |
| define-epics | `define-epics/SKILL.md` | epic preferences, directory name, approval |
| execute-plan | `execute-plan/SKILL.md` | checkpoint approvals (proceed, approve plan, approve build) |

## The Live Responder

Monitors Claude's stdout during a test run and responds when Claude asks something the setup agent's prompt didn't anticipate.

**Detection heuristics for "Claude is waiting for input":**

- Output has stopped for >5 seconds
- Last line ends with `?` or contains a multiple-choice prompt
- Claude printed an `AskUserQuestion`-style list

When triggered, calls haiku with the last Claude turn + analyst context to generate a natural analyst response, then writes it to the subprocess's stdin.

**Guardrails:**

- Max 20 responses per phase
- Configurable phase timeout (default 10 minutes)
- Passthrough mode for execute-plan checkpoints (just says "yes, proceed")
- Full transcript logged to `integration_tests/.logs/<phase>-<timestamp>.txt`

**Fallback:** If live detection proves fragile, fall back to `--print` mode with the full prompt (losing surprise handling but gaining reliability).

## Phase Tests and Assertions

### Happy-path tests

**`test_init.py`**

| Structural (hard gate) | Quality (advisory) |
|---|---|
| `CLAUDE.md` exists at project root | `CLAUDE.md` references the project name |
| `kyros-agent-workflow/` directory exists | `.harnessrc` has all config sections |
| `.harnessrc` exists and parses as YAML | `docs/standards/` files have non-empty content |
| `docs/standards/` contains coding-standards, definition-of-done, review-criteria | |
| `docs/solutions/` directory exists | |

**`test_profile_data.py`**

| Structural | Quality |
|---|---|
| `docs/context/data-profile-customers.md` exists | Profile mentions all 6 column names |
| `docs/context/analyst-notes.md` exists | Profile contains distribution stats or null rates |
| Both files are non-empty markdown | Analyst notes reference "churned" as target |

**`test_define_epics.py`**

| Structural | Quality |
|---|---|
| Epics directory exists at the expected path | Each epic YAML has `acceptance_criteria` field |
| At least 2 epic YAML files created | Epics are sequentially numbered (01-, 02-, ...) |
| Each file parses as valid YAML | Epic names relate to the modeling task |

**`test_execute_plan.py`**

| Structural | Quality |
|---|---|
| Epic branch `epic/<name>` created | Tests are actual pytest files with assertions |
| `.execution-state.yaml` exists and parses | Implementation plan has one step per test file |
| Git tag `tdd-baseline-<epic>` exists | PR description summarizes work done |
| Tests exist before implementation code (tag check) | Solution docs (if any) have valid frontmatter |
| All tests pass at end of build | |
| PR created on the repo | |

### Assertion runner

Both tiers share a common interface:

```python
class AssertionResult:
    passed: bool
    message: str
    tier: Literal["structural", "quality"]

def check_phase(project_dir: Path, phase: str) -> list[AssertionResult]:
    results = structural.check(project_dir, phase)
    results += quality.check(project_dir, phase)

    hard_failures = [r for r in results if not r.passed and r.tier == "structural"]
    warnings = [r for r in results if not r.passed and r.tier == "quality"]
    # Hard failures fail the test; warnings are logged
```

### Edge-case tests

**`test_circuit_breakers.py`**

| Test | How induced | Assertion |
|---|---|---|
| `test_no_progress_halts` | Epic with impossible acceptance criteria (99.9% accuracy) | Execution state shows `status: blocked`, circuit breaker reason logged, halts within `no_progress_threshold` iterations |
| `test_repeated_error_halts` | Inject broken dependency (`import nonexistent_lib`) | Halts after `same_error_threshold` repetitions, error context captured |
| `test_review_rounds_exceeded` | Deliberately vague acceptance criterion ("code must be elegant") | Halts after `max_review_rounds`, escalation message shown |

**`test_resume_recovery.py`**

| Test | How induced | Assertion |
|---|---|---|
| `test_resume_after_interrupt` | Kill subprocess mid-build, restart `/execute-plan` | Offers resume, resumes from correct step, completed steps not re-run |
| `test_dod_failure_autofix` | Inject `TODO` comment and `print("DEBUG")` after build | DoD catches violations, auto-fix removes them, PR created |
| `test_quality_scan_advisory` | Inject unused import after build | Quality scan flags in PR description, does not block PR creation |

**`test_guards.py`**

| Test | How induced | Assertion |
|---|---|---|
| `test_missing_epics_dir` | `/execute-plan nonexistent/path` | Clear error, no crash, no state file |
| `test_malformed_harnessrc` | Invalid YAML in `.harnessrc` | Config error reported before execution |
| `test_concurrent_execution_warning` | Fake `.execution-state.yaml` with `status: in_progress` | Warning surfaced |
| `test_empty_data_file` | `/profile-data` against headers-only CSV | Handles gracefully |

Edge-case fixtures use low circuit breaker thresholds (e.g. `no_progress_threshold: 2`) and tight timeouts to keep test runs fast. Edge-case tests run in parallel via pytest-xdist.

## The Fix Agent

Triggered by any structural assertion failure. Quality warnings do not trigger it.

### Fix cycle

1. **Collect failure context:** failed test name and assertions, full Claude session transcript, test project directory state, relevant plugin source files (skill, hooks, templates).

2. **Launch fix session:** New `claude` subprocess pointed at the plugin repo with a structured diagnostic prompt including the failure context. Instructed to diagnose and fix the plugin code.

3. **Guardrails:**
   - Can only modify files under `skills/`, `commands/`, `hooks/`, `templates/`, `agents/`, and `lib/`. Never the test suite.
   - Max 5 minutes per fix attempt.
   - Changes land on branch `fix/integration-<phase>-<timestamp>` for human review.

4. **Re-run the test** against the fixed plugin code.

5. **Max 3 fix-then-retry cycles** per test. After 3 failures: marked `FAILED_AFTER_RETRIES`, fix branches preserved.

## Test Runner and Orchestration

`run_all.py` is the entry point.

### Execution flow

```
run_all.py
  │
  ├─ 1. Regenerate synthetic data (or verify cached CSV matches)
  │
  ├─ 2. Run happy-path tests in phase order:
  │     ├─ test_init
  │     ├─ test_profile_data
  │     ├─ test_define_epics
  │     └─ test_execute_plan
  │
  ├─ 3. Run edge-case tests (parallel via pytest-xdist):
  │     ├─ test_circuit_breakers
  │     ├─ test_resume_recovery
  │     └─ test_guards
  │
  ├─ 4. For each failure: fix-then-retry loop (up to 3 attempts)
  │
  └─ 5. Generate report
```

### CLI interface

```bash
# Run everything
python integration_tests/run_all.py

# Run a single phase
python integration_tests/run_all.py --phase profile-data

# Run only edge cases
python integration_tests/run_all.py --edge-cases-only

# Run chained (each phase feeds into the next, no fixtures)
python integration_tests/run_all.py --chained

# Skip the fix loop (just report failures)
python integration_tests/run_all.py --no-fix

# Set custom timeouts
python integration_tests/run_all.py --phase-timeout 600 --fix-timeout 300
```

### Chained mode

A single temp directory is shared across all happy-path phases. Output from `/init` feeds into `/profile-data`, and so on. If a phase fails, subsequent phases are skipped. The fix loop runs on the failing phase; if the fix succeeds, the chain resumes.

### Cost tracking

Each test run logs token usage to `.reports/cost-history.json`. `run_all.py` prints an estimated cost at the start based on prior runs and actual usage after each test.
